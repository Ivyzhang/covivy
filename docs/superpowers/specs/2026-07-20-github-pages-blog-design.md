# Covivy GitHub Pages 博客设计

## 目标

创作一篇容易阅读的中文推荐文章：既让开发者和工程团队愿意尝试 Covivy，也让招聘者看见项目背后的工程判断与技术深度。

## 目标读者

- 不满足于单个项目总覆盖率数字，希望获得 Pull Request 级别可执行反馈的开发者。
- 正在评估自托管覆盖率服务的技术负责人和 DevOps 工程师。
- 希望了解作者在后端、基础设施、第三方集成和编程语言工具链方面能力的招聘者与面试官。

## 写作方式

文章采用故事型技术案例结构。开头先展示“项目总覆盖率很高，新代码却没有测试”的矛盾，再跟随一份覆盖率报告走完整个 Covivy 流程。技术细节只在解释实际问题时出现，避免写成 README 或功能清单。

幽默来自开发者熟悉的真实场景和克制的比喻，不牺牲技术可信度。核心比喻是：项目总覆盖率像全班平均分，patch coverage 则专门检查这次交作业的人有没有认真完成。文章不使用无法证实的性能、用户量和稳定性数据。

## 内容结构

1. 冲突式开场：总覆盖率可能掩盖没有测试的新代码。
2. Covivy 是什么，以及为什么应该把覆盖率带到 PR 上。
3. 用通俗语言讲清从 CI 上传到 PR 反馈的完整链路。
4. 核心差异：语义化变更行分析、多种报告格式归一化、文件和行级可执行反馈。
5. 工程深度：异步 Worker、PostgreSQL 行锁与重试、webhook 和 upload 乱序、force-push 旧报告隔离、GitHub App、token 权限边界与 OAuth 恢复。
6. 诚实说明边界：GitHub 已具备完整端到端链路；GitLab 当前重点是 OAuth、仓库发现和 provider 基础能力。
7. 适合谁使用、精简快速开始，以及面向招聘者的“这个项目证明了什么”。
8. 仓库入口和 GitHub Pages 发布步骤。

## 页面设计

- 正文文件：`docs/index.md`，由 GitHub Pages/Jekyll 渲染。
- 配置文件：`docs/_config.yml`，使用克制、易读的主题和仓库元数据。
- 图片：`docs/assets/covivy-flow.png`，用多种克制颜色展示简洁的架构流程。
- 排版：短段落、描述性标题、少量提示块，只在有助于比较时使用表格，并提供可直接执行的命令。
- 不使用营销式大 Hero、动画、超大字号、嵌套卡片或堆砌功能说明的版式。

## 准确性边界

- 只描述能够从仓库代码和文档中验证的行为。
- 明确 GitHub 是当前完整的端到端反馈路径。
- 明确 GitLab 已支持 OAuth、仓库发现和 provider 基础能力，但尚未形成完整的异步 Merge Request 反馈链路。
- 使用项目实际支持的 Python 3.9+ 和现有本地启动步骤。
- 不把 Covivy 描述成无需运维的托管 SaaS，也不声称它适合所有生产环境。

## 验证方式

- 对照仓库源码检查命令、路径、链接和功能描述。
- 检查 Markdown 结构和 Jekyll 配置语法。
- 生成 PNG 架构图并检查尺寸、文字和内容。
- 在环境支持时本地启动 `docs` 页面，检查桌面与移动端阅读效果。
- 如果只修改文档，不必重复测试所有业务代码；运行文档专项检查和现有 lint，确保没有引入无关回归。
